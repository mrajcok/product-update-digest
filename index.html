How to build an AI telemetry architecture that serves five teams without duplicate collection | Cribl

Cribl introduces an architecture for AI telemetry that enables a single collection pass for multiple teams, avoiding redundant data collection and associated costs.

  • The architecture normalizes heterogeneous schemas from sources like OpenTelemetry GenAI and GPU metrics (DCGM, vLLM, Triton, TensorRT-LLM) into a consistent internal schema.
  • Cribl Stream and Edge are utilized for ingestion: Stream handles OTLP and API sources, while Edge handles on-premises GPU clusters with local data redaction and enrichment.
  • In-flight policy features include PII redaction, workload enrichment (attaching metadata like team and model version to GPU metrics), and cost metric derivation (calculating per-request costs).
  • Data is routed to various destinations (SRE/APM, Security/SIEM, FinOps, Eval platforms, Cribl Lake) at the appropriate fidelity and cost tier, with unified querying across all destinations provided by Cribl Search’s lakehouse and federated engines.

Why agentic analytics breaks the modern data stack — and how OcientAIQ™ is built for what's next

OcientAIQ™ is designed to address the limitations of the modern data stack when used with agentic analytics.

  • OcientAIQ™’s pricing is fixed per-core and capacity-based, removing cost constraints on agent exploration and iteration; further details are available at docs.ocient.com.

Cribl Search: AI-Powered Log Investigation

Cribl Search offers faster log investigations without the costs and complexity of a SIEM, leveraging AI to accelerate workflows and reduce manual toil.

  • Unified Platform: Combines data collection, normalization, storage, search, dashboards, and alerting into a single product.
  • AI-Assisted Investigation: Uses agentic AI and natural language queries to guide investigations, reducing the need for complex query building and accelerating time-to-resolution.
  • Search-in-Place Architecture: Eliminates data rehydration by searching data in its existing location (data lakes, object stores, etc.), reducing costs and delays.
  • Search Packs: Pre-built dashboards, searches, and configurations for common use cases (e.g., AWS WAF, Kubernetes) are available to accelerate onboarding and provide instant visibility.

Cribl Lake - Your Scalable Cloud Data Lake Solution | Cribl

Cribl Lake provides a scalable cloud data lake solution designed for telemetry data, emphasizing ease of use and cost reduction.

  • Stores any type of telemetry data in raw, structured, or unstructured formats, supporting tiered storage based on data value and usage.

Ocient blog · 2026-06-02

Graph analytics at scale without Spark bottlenecks

OcientAIQ™ introduces OCGraph, a library for running large-scale graph algorithms directly within the platform, avoiding Spark bottlenecks.

  • OCGraph is available in Java (via the Ocient JDBC driver – com.ocient:ocient-jdbc4) and Python (pip install ocient-graph) with identical APIs.
  • Graphs are represented as two relational tables: a vertices table (with an id column) and an edges table (with srcid and destid columns). This allows leveraging existing Ocient table features like indexing and joining.
  • The library currently supports algorithms including Connected Components, PageRank, Triangle Count, Weighted Shortest Paths, and Label Propagation, offering similar functionality to GraphX/GraphFrames but with a simplified API.
  • Benchmarking against GraphFrames/GraphX on the LDBC Graphalytics twitter_mpi dataset (52.6M vertices, 1.96B edges) showed OCGraph running algorithms 7x to 17x faster, and successfully completing Triangle Count where Spark failed.

Data Lake vs. Data Warehouse: What is the Difference | Cribl

Data Lake vs. Data Warehouse: What is the Difference?

This article discusses the evolving landscape of data storage and processing, particularly in the context of AI and machine learning. It highlights that the choice isn't simply between data lakes and data warehouses, but building a comprehensive data architecture that emphasizes data quality, governance, and pipeline control.

  • Data lakes store raw, multi-format data cheaply, ideal for ML, telemetry, observability, and exploratory analysis.
  • Data warehouses store cleaned, structured data optimized for fast BI, reporting, and governed analytics.
  • Lakehouses combine the scale and flexibility of lakes with the performance and governance of warehouses, leading to hybrid architectures.
  • Cribl helps teams filter, enrich, route, and reduce telemetry data before storage to control costs and complexity – with features like Cribl Lake and Cribl Search.

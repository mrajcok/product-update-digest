Graph analytics at scale without Spark bottlenecks

OcientAIQ™ now features OCGraph, a library for running large-scale graph algorithms directly within the platform, eliminating Spark bottlenecks.

  • OCGraph is available in Java (as part of the Ocient JDBC driver – com.ocient:ocient-jdbc4) and Python (ocient-graph via PyPI). Both libraries offer identical sets of algorithms.
  • Graphs are represented as two standard Ocient tables: a vertices table (with 'id' as a required column) and an edges table (with 'srcid' and 'destid' as required columns). This allows leveraging existing Ocient table features like indexing and joins.
  • The library includes algorithms such as Connected Components, PageRank, Triangle Count, Weighted Shortest Paths, and Label Propagation, offering comparable functionality to Spark’s GraphX/GraphFrames with a simplified API.
  • Benchmarks on the LDBC Graphalytics XL twitter_mpi dataset (52.6M vertices, 1.96B edges) show OCGraph achieving performance improvements of 7x to 17x for iterative algorithms like PageRank, Connected Components, and Label Propagation, with successful (and faster) completion of triangle count where Spark failed.

Inside the Ocient Engine: Casting, timestamps, and “secret” internal names

Ocient provides flexible SQL syntax options, drawing from Postgres and DB2, with internal optimizations revealed through generated column names.

  • Ocient supports multiple functions for obtaining the current timestamp—now(), current_timestamp, and standard scalar functions—all mapping to the same internal function, distinguished by appended suffixes.
  • Casting can be performed using ANSI SQL, Postgres short syntax (::), explicit CAST functions, or by invoking the target data type as a function (constructor style).
  • Intervals are internally represented as integer/bigint values with associated units, and constructing intervals is preferred over casting to an INTERVAL type. Timestamp subtraction yields bigint values in nanoseconds, while date subtraction results in integer days.
  • Ocient offers both standard EXTRACT and date_part functions (returning doubles) alongside shorthand functions like year() and minute() (returning integers/bigints) for extracting date parts.

Inside the Ocient Engine: A deep dive into arrays

Ocient’s array functionality treats arrays as a core data model component, supporting complex nesting and high-performance indexing.

  • Ocient infers array types (e.g., BIGINT) but allows explicit casting for control, even handling arrays of purely NULL values which default to CHAR but can be overridden.
  • The platform supports nested and jagged (variable-length) multi-dimensional arrays.
  • Ocient provides array operators like @> (contains) and && (overlap) and introduces unique quantifiers for_some and for_all for expressive filtering without unnesting.
  • Indexing is supported on array columns, including N-gram indexes for text arrays, and arrays can be constructed dynamically using array_agg with enforced ordering; unnest offers both standard explosion and "Zip Unnest" for aligned data.

Compute-Adjacent Storage at Work: 24 drives, 9.6 TB, 1 query

OcientAIQ demonstrates high sustained I/O throughput through its “compute-adjacent storage” model, pushing processing to the drive level.

  • A benchmark utilizing 24 NVMe drives (3 nodes of 8 drives each) processed 9.6 TB of uncompressed, randomly generated 64-bit integers.
  • A single query achieved aggregate throughput of 35.82 GB/s, or 1.49 GB/s per drive, sustained for 268 seconds.
  • This performance represents 29-44% of the theoretical maximum read speed of the Samsung NVMe drives used (rated at 5.2 GB/s sequential reads).
  • The architecture avoids network bandwidth bottlenecks by filtering and processing data directly at the drive level, only propagating a single count value upwards.

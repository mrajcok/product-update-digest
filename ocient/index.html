Graph analytics at scale without Spark bottlenecks

OcientAIQ™ introduces OCGraph, a library for running large-scale graph algorithms directly within the platform, avoiding Spark bottlenecks.

  • OCGraph is available in Java (via the Ocient JDBC driver - com.ocient:ocient-jdbc4) and Python (ocient-graph via PyPI). Both libraries offer identical algorithms and semantics.
  • Graphs are represented as two relational tables: a vertices table (with id as a primary key) and an edges table (with srcid and destid defining relationships). This leverages existing Ocient table features like indexing and querying.
  • The library provides algorithms including Connected Components, PageRank, Triangle Count, Weighted Shortest Paths, and Label Propagation, mirroring functionalities found in GraphX and GraphFrames but without the associated complexity.
  • Benchmarking against the LDBC Graphalytics dataset shows OCGraph performing 7x - 17x faster than Spark for iterative algorithms, and successfully completing Triangle Count where Spark failed due to memory limitations.

Inside the Ocient Engine: Casting, timestamps, and “secret” internal names

Ocient SQL offers flexible casting and timestamp handling, drawing from Postgres and DB2 concepts with unique internal optimizations.

  • Ocient supports multiple functions for retrieving the current timestamp—now(), current_timestamp, and scalar functions—all mapping to the same internal function, distinguished by appended suffixes (e.g., _1, _2).
  • Casting can be performed using ANSI SQL, Postgres-style ::, explicit CAST functions, or constructor-style casts, where data type names act as functions (e.g., CHAR()).
  • Intervals are internally represented as integer or bigint values with assigned units, meaning INTERVAL is not a native data type, and date math is best handled using constructors. Subtracting timestamps returns bigint values in nanoseconds, while subtracting dates returns integers representing days.
  • Ocient provides both standard EXTRACT and date_part functions (returning doubles) alongside shorthand functions like year() and minute() (returning integers/bigints); shorthand functions extract specific portions of a timestamp.

Inside the Ocient Engine: A deep dive into arrays

Ocient’s engine treats arrays as a core data model component, supporting features beyond typical database implementations.

  • Type Inference & Casting: Ocient automatically infers array data types but allows explicit casting, including handling arrays containing only NULL values (defaulting to CHAR, but allowing forced type specification).
  • Complex Array Structures: The platform supports nested, multi-dimensional, and jagged arrays—arrays where sub-arrays have differing lengths.
  • Array Operators: Ocient provides standard array operators like @> (Contains) and && (Overlap), as well as unique “Ocient Special” quantifiers: for_some and for_all for applying scalar predicates across array elements without unnesting.
  • Performance & Manipulation: Users can create indexes on array columns (including N-gram indexes for text), aggregate data into arrays with controlled order using array_agg, and utilize unnest with standard and "Zip Unnest" functionality to manage array expansion and alignment.

Inside the Ocient Engine: A deep dive into tuples

Ocient’s tuple data type functions as an anonymous struct, enabling powerful querying techniques within the OcientAIQ™ Unified Data Platform.

  • Tuples are defined using double angle brackets (<<...>>) for type specification, similar to C++ templates or Java generics, and are accessed by position (starting at 1).
  • Ocient stores tuples as either virtual (separate columns) or materialized (binary blob) structures; tuples within arrays are always materialized, while others are virtualized on disk and may be materialized during query execution.
  • Tuple comparisons are performed lexicographically, influencing sorting and grouping operations, and tuples can be nested with other tuples and arrays.
  • Indexing tuples requires indexing specific elements rather than the entire tuple, and a "Tuple Max" trick can efficiently handle point-in-time queries on append-only tables using MAX() and tuple nesting.
